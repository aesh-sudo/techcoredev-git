Файл деплоймента ```app-deployment.yml```:
```
apiVersion: apps/v1                        1
kind: Deployment                           2
metadata:                                  3
  name: my-app-deployment
spec:                                      4
  replicas: 3                              5
  selector:                                6
    matchLabels:                           7
      app: my-app
  template:                                8
    metadata:                              9
      labels:                             10
        app: my-app
    spec:                                 11
      containers:                         12
      - name: my-app                      13
        image: my-app_12:k8s              14
        imagePullPolicy: IfNotPresent     15
```

```1``` – версия API k8s. С развитием «k8s» разные типы ресурсов появляются в разных версиях API. ```Deployment``` - это ресурс из группы ```apps```, стабильная версия которого – ```v1```.

```2``` – тип создаваемого ресурса.

```3``` – метаданные объекта ```deployment```:<br>
```name``` – имя объекта.

```4``` – раздел описания желаемого состояния объекта.

```5``` – количество копий (реплик) приложения, которые деплоймент должен поддерживать.

```6``` – механизм, с помощью которого деплоймент отбирает только свои поды среди подов в кластере.<br>
Метки в ```template``` (поде) и метки в ```selector``` (отборщик) должны совпадать.

```7``` – тип селектора. Есть более мощный тип селектора – ```matchExpressions```.<br>
Пример:
```
selector:
  matchExpressions:
  - key: app
    operator: In
    values:
    - my-app
    - other-app
```
Описание: ищи поды, у которых метка ```app``` имеет значение ```my-app``` или ```other-app```.<br>
Оператор (```operator```) может быть другой:<br>
```In``` – значение есть в списке.<br>
```NotIn``` – значения нет в списке.<br>
```Exists``` – метка существует (любое значение).<br>
```DoesNotExist``` – метки нет.

Правило:<br>
Метки из ```selector``` должны быть в ```template```. Обратное – не обязательно.<br>
Пример:
```
selector:
  matchLabels:
    app: my-app
    version: v1
template:
  metadata:
    labels:
      app: my-app
```
Деплоймент будет искать поды с меткой ```version: v1```, но создаст поды с другой меткой. В результате он никогда не найден свои поды и будет бесконечно создавать новые.

```8``` – шаблон, по которому деплоймент создает поды.<br>
Все, что внутри ```template```, описывает один под. Далее деплоймент по этому шаблону создаст три пода.

```9``` – метаданные пода.

```10``` – метки пода. Пары ```ключ:значение```, присваиваемые объектам «k8s» для их идентификации и поиска.

```11``` – описание того, из чего состоит под. 

```12``` – список контейнеров в поде.

```13``` – имя контейнера. По имени можно обращаться к контейнеру в поде (например, для просмотра логов.)

```14``` – Docker-образ контейнера.

```15``` – политика загрузки образа. Параметр может принимать одно из значений:<br>
```IfNotPresent``` – образ доступен локально, скачивать не нужно.<br>
```Always``` – всегда скачивать из «registry».<br>
```Never``` – никогда не скачивать. Только локальные образы.

Отправим файл в «minikube» для обработки:
```bash
$ kubectl apply -f app-deployment.yml
deployment.apps/my-app-deployment created
```

Проверим, что создал «minikube»:
```bash
$ kubectl get pods -l app=my-app
NAME                                 READY   STATUS    RESTARTS   AGE
my-app-deployment-7685f5c785-9c85s   1/1     Running   0          6s
my-app-deployment-7685f5c785-r5dwn   1/1     Running   0          6s
my-app-deployment-7685f5c785-x8qhc   1/1     Running   0          6s
```

Удалим один из подов:
```bash
$ kubectl delete pod my-app-deployment-7685f5c785-x8qhc
pod "my-app-deployment-7685f5c785-x8qhc" deleted from default namespace

$ kubectl get pods -l app=my-app
NAME                                 READY   STATUS    RESTARTS   AGE
my-app-deployment-7685f5c785-9c85s   1/1     Running   0          100m
my-app-deployment-7685f5c785-r5dwn   1/1     Running   0          100m
my-app-deployment-7685f5c785-wxprz   1/1     Running   0          35s
```

Видно, что удаленный под исчез и сразу появился новый под (с другим именем).

Команда для наблюдения за подами в реальном времени:
```bash
kubectl get pods -l app=my-app -w
```

Создадим ```Service``` - файл «app-service.yml»:
```
apiVersion: v1             1
kind: Service              2
metadata:
  name: my-app-service     3
spec:
  selector:
    app: my-app            4
  ports:                   5
  - port: 80               6
    targetPort: 5000       7
  type: NodePort           8
```
```1``` – версия API. ```Service``` - базовый ресурс «k8s», который находится в основной группе API (```v1```).<br>
```2``` – тип ресурса.<br>
```3``` – имя сервиса. По имени поды могут обращаться к сервису.<br>
```4``` – связь сервиса с подами. Сервис находит все поды с меткой ```app: my-app``` и направляет трафик на них.<br>
```5``` – список портов. Сервис может слушать несколько портов. В примере – один.<br>
```6``` – порт, на котором сервис слушает внутри кластера. Другой под может обратиться к приложению так:<br>
```http://my-app-service:80```<br>
```7``` – порт приложения внутри пода. Приложение внутри контейнера слушает порт ```5000```. Сервис получает запрос на порт ```80``` и перенаправляет его на порт ```5000``` в поде.<br>
```8``` – тип сервиса. По умолчанию сервис имеет тип ```ClusterIP``` - доступен только внутри кластера.<br>
Нам нужно открыть приложение снаружи (в браузере).<br>
```NodePort``` выполняет два действия:
* Выделяет порт в диапазоне ```30000-32767``` на каждой ноде (в «minikube»).
* Перенаправляет трафик с этого порта на сервис.

Отправим файл сервиса в «minikube» для обработки:
```bash
$ kubectl apply -f app-service.yml
service/my-app-service created
```
Чтобы проверить работу приложения в браузере хоста, на котором развернута VM, необходимо выполнить команду:
```bash
nohup kubectl port-forward svc/my-app-service 8080:80 --address 0.0.0.0 > port-forward.log 2>&1 &
```

```nohub``` – поддерживает работу процесса даже после выхода из терминала.<br>
```&``` (в конце команды) – запуск процесса в фоновом режиме.<br>
```port-forward``` – команда создает безопасный туннель между локальным портом и портом ресурса в кластере.<br>
```svc/``` - путь к сервису.<br>
```8080:80``` – маппинг портов. Локальный порт – ```8080```. Порт сервиса – ```80```.<br>
```--address 0.0.0.0``` – прием соединений на всех доступных сетевых интерфейсах.<br>
```> port-forward.log``` – перенаправление стандартного вывода в файл.

Чтобы удалить туннель, необходимо найти и убить его процесс:
```bash
ps aux | grep port-forward
kill <PID>
```

В браузере приложение будет доступно по адресу:
```
http://адрес_хоста:8080
```
